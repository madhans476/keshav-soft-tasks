# Orbiq — Bootstrap 5 Website (Internship Task 1)

A 3-page responsive marketing site (`index.html`, `about.html`, `contact.html`) built with HTML5, CSS3, and Bootstrap 5 via CDN, for a fictional workflow-automation product called **Orbiq**.

This README is a transparent account of how the project was actually built, so that anyone reviewing it — a mentor, a reviewer, or future-me — knows exactly what was done, what tools were involved, and where the AI's contribution starts and stops. Nothing below is dressed up.

---

## 1. How this was built, step by step

1. **Defined the brief.** Requirements were: 3 pages (Home, About, Contact), Bootstrap 5 via CDN only, original combination of components (not a copied Bootstrap template), consistent navbar/footer, a features/testimonials/stats section, a services/team/timeline section, and a validated contact form with FAQ accordion and map embed.
2. **Picked a concrete subject.** Rather than filling the template with generic "Lorem ipsum" business copy, a specific fictional product (Orbiq, a workflow-automation SaaS) was invented so every section — hero, features, testimonials, FAQ — has real, consistent copy instead of placeholder text.
3. **Designed a small token system before writing code**: a named color palette (indigo `#2B4FFF` + amber `#FFB020` on white/light-grey), a type pairing (Space Grotesk for headings, Inter for body, IBM Plex Mono for small uppercase labels), and one signature visual (an animated CSS/SVG "orbit ring" in the hero) — deliberately chosen to avoid the generic AI-website look (cream + terracotta, or dark + neon).
4. **Built the shared CSS file first** (`css/style.css`) — design tokens as CSS variables, then the hero animation, then card/section/footer refinements — so all three pages could reuse it consistently.
5. **Built each page**, reusing the same navbar and footer markup across all three, and marking the correct nav link `active` per page.
6. **Added interactivity**: Bootstrap's `needs-validation` client-side form validation on the contact form, a working Bootstrap accordion for FAQs, and a Google Maps `iframe` embed (no API key required).
7. **Checked the output**: ran the HTML through a parser (BeautifulSoup) to confirm every page parses without broken/unclosed tags, and did a manual pass for consistent spacing, alt text on every image, and keyboard-focus visibility.
8. **Packaged the deliverable**: `index.html`, `about.html`, `contact.html`, and `css/style.css` in one folder, ready to open directly in a browser (no build step, no npm install — everything is CDN-based).

## 2. Tools and resources used — full disclosure

**This project was built with substantial AI assistance from Claude (Anthropic).** To be direct about what that means:

- The **HTML, CSS, and copywriting were generated by Claude** inside a single chat conversation, based on a detailed prompt specifying the required pages, sections, and constraints.
- **Bootstrap 5, Bootstrap Icons, and Google Fonts** were pulled from their official CDNs — no Bootstrap example page was copied wholesale; components (navbar, cards, accordion, forms, grid) were selected and combined based on the brief.
- **AI tool was used** in this instance — Claude was the sole AI assistant involved, used directly through this conversation.
- **No code was copied from Stack Overflow, CodePen, or third-party templates.** The one exception is the *use* of standard, publicly documented Bootstrap component markup (e.g. the shape of a `navbar-expand-lg`, or an `accordion` block) as documented on getbootstrap.com — which is expected and normal when working with a component framework, not "copying a project."
- Bootstrap's **official documentation** (getbootstrap.com) is the implicit reference for how components like the navbar, cards, forms, and accordion are structured; no external tutorials or templates were consulted.

**What this means in practice:** if this is submitted as an internship deliverable, it should be described honestly as *AI-assisted development* — the code was authored by an AI model based on a human-written specification, not typed line-by-line by a human learning Bootstrap through trial and error. If the internship expects hand-written code as a learning exercise, this project in its current form would not reflect that, and that's worth flagging to whoever is reviewing it rather than presenting it as unaided work.

If the goal is to *learn* Bootstrap rather than to *ship* a page, a genuinely more valuable next step would be to rebuild one section (e.g. just the navbar and hero) by hand using this file as a reference, then compare — that would produce real, defensible learning to describe in a report.

## 3. Challenges faced (technical, in this build)

Being honest here too: because this was generated by an AI assistant in one sitting rather than debugged by a human over multiple attempts, the "challenges" are of a different nature than a typical learner would face. The real friction points were:

- **Avoiding a templated, default-Bootstrap look.** The first instinct for any AI-generated Bootstrap site is a stock navbar + generic 3-card row + basic footer. Extra thought went into a distinct color/type system and one custom animated element (the orbit ring) so the site wouldn't look like an out-of-the-box Bootstrap starter.
- **Keeping the "Why choose us" section meaningful rather than decorative.** Numbered steps (01–04) are only honest when they represent a real sequence — so that section was deliberately built as an actual onboarding process (Discover → Integrate → Automate → Scale) rather than four arbitrary numbered cards.
- **Form validation without a backend.** Since there's no server to actually receive the contact form, Bootstrap's `needs-validation` pattern was wired up with a small vanilla-JS snippet that intercepts `submit`, checks `checkValidity()`, and shows a success alert — clearly commented as a placeholder for a real backend/API call.
- **Cross-page consistency.** With three separate HTML files (no templating engine), the navbar and footer markup had to be kept identical across pages by hand, only changing the `active` class per page — a real risk of drift that a human maintaining this by hand should watch for if the pages are edited later.
